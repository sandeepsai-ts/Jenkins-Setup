Before You Start

You need: a Mac or Windows machine with admin access, a Testsigma account with at least one Test Plan already set up, and 30–45 uninterrupted minutes. You do not need to already know Groovy, Jenkins, or shell/PowerShell scripting — just follow the steps in order for whichever OS you're on (use the toggle above).

This builds a Jenkins job that triggers a Testsigma Test Plan on a schedule, waits for it to actually finish, fetches the results, and turns them into one readable HTML report — automatically, every run, with no manual steps in between. Here's the shape of it:

If you ever click Abort partway through, Testsigma is told to stop before the report step runs, so the report reflects what actually happened rather than a stale in-progress snapshot (Part 9).

Nothing in these scripts or in the Jenkinsfile is tied to one person's machine. Every path is built from environment variables that Jenkins fills in automatically ($(whoami) / $env:USERNAME, env.HOME / env.USERPROFILE, env.JOB_NAME), so this guide works as-is on any Mac or Windows agent. Anywhere you'd normally have to hunt down and edit a hardcoded value, you'll instead see a 🔧 Customizable flag telling you exactly what it is and where it lives.
What you get out of the box: a 4-hour failsafe so a stuck Test Plan can't hang the build forever (Part 10); Suite, Case, and Step results fetched automatically with zero manual ID lookup (Part 10); a single self-contained HTML report that opens offline, with the Test Plan Summary leading in plain English and everything else one click away (Part 11); and an Abort button that actually frees your Testsigma license instead of leaving the run going in the background (Part 9). The only thing you ever choose per build is how long to wait before giving up on a stuck run — everything else just works.
Already using an earlier copy of this guide? A few things changed worth knowing before you re-paste anything: the "Build with Parameters" screen now asks for only one value (the failsafe timeout) instead of several — the old per-device (MACHINE) result level was dropped entirely, and Suite/Case ID fields are gone since results are found automatically now. There was also a real bug in older polling scripts where a successful run could poll forever instead of finishing, and a Mac-only crash in older fetch scripts right after auto-discovery found results — both are fixed here. Part 12's Reference section has the full technical explanation for each if you want it; the short version is: update your scripts and Jenkinsfile to the copies in this guide and you're current.

Part 3 — One Thing to Know Before Continuing

Jenkins has an official Testsigma plugin. This guide does not use it, on purpose — it has an unresolved security issue (Jenkins Security Advisory 2025-07-09, CVE-2025-53661: it doesn't hide your Testsigma API key properly) and hasn't been updated since 2022. This setup uses plain scripts (shell on Mac, PowerShell on Windows) and Jenkins' built-in Pipeline feature instead — nothing Testsigma-specific to install here, on either OS.

#!/bin/bash
# -------------------------------------------------------------
# trigger-run.sh
# Triggers a Testsigma execution and saves the Run ID to a
# fixed path in /tmp so the Jenkins abort hook can always find
# it regardless of workspace state.
#
# Required environment variables (injected by Jenkins):
#   TS_API_TOKEN  -- Testsigma API key (from Jenkins Credential TS_API_KEY)
#   TS_PLAN_ID    -- Execution ID from Testsigma CI/CD Integrations tab (from Jenkins Credential TS_PLAN_ID)
#   APP_VERSION   -- passed as buildNo for traceability in Testsigma UI
#   RUN_ID_FILE   -- fixed path set in the Jenkinsfile environment block
# -------------------------------------------------------------

set -euo pipefail

if [[ -z "${TS_PLAN_ID:-}" || -z "${TS_API_TOKEN:-}" ]]; then
  echo "[ERROR] TS_PLAN_ID and TS_API_TOKEN must be set."
  exit 1
fi

echo "[INFO] Triggering plan ID: ${TS_PLAN_ID} | Version: ${APP_VERSION:-unset}"

TMPFILE=$(mktemp)
HTTP_CODE=$(curl -s -o "$TMPFILE" -w "%{http_code}" \
  -X POST \
  -H "Content-Type: application/json" \
  -H "Accept: application/json" \
  -H "Authorization: Bearer ${TS_API_TOKEN}" \
  "https://app.testsigma.com/api/v1/execution_results" \
  -d "{\"executionId\": \"${TS_PLAN_ID}\", \"buildNo\": \"${APP_VERSION:-manual}\"}")

BODY=$(cat "$TMPFILE")
rm -f "$TMPFILE"

if [[ "$HTTP_CODE" != "200" && "$HTTP_CODE" != "201" ]]; then
  echo "[ERROR] Testsigma returned HTTP ${HTTP_CODE}"
  echo "$BODY"
  exit 1
fi

RUN_ID=$(echo "$BODY" | python3 -c "import sys,json; print(json.load(sys.stdin)['id'])")

if [[ -z "$RUN_ID" || "$RUN_ID" == "null" ]]; then
  echo "[ERROR] Could not read Run ID from response."
  echo "$BODY"
  exit 1
fi

# Write to the fixed path declared in the Jenkinsfile -- survives
# workspace cleanup and Jenkins abort. The fallback below only
# applies if you run this script manually, outside Jenkins.
echo "$RUN_ID" > "${RUN_ID_FILE:-/tmp/ts_run_id_manual-run}"
echo "[OK] Execution started. Run ID: ${RUN_ID}"
echo "[INFO] Run ID saved to: ${RUN_ID_FILE:-/tmp/ts_run_id_manual-run}"

#!/bin/bash
# -------------------------------------------------------------
# poll-status.sh
# Polls Testsigma until the run reaches a terminal status, or
# (optionally) a safety-cap timeout elapses -- whichever comes
# first. The safety cap is togglable via MAX_WAIT_MINUTES without
# editing this script.
#
# Required environment variables (injected by Jenkins):
#   TS_API_TOKEN  -- Testsigma API key
#   RUN_ID_FILE   -- fixed path set in the Jenkinsfile environment block
#
# Optional environment variables:
#   MAX_WAIT_MINUTES        -- safety-cap timeout in minutes.
#                              Unset, empty, "0", or "unbounded"/"none"
#                              (case-insensitive) DISABLES the cap --
#                              polling continues purely on terminal-status
#                              detection. Any positive integer ENFORCES
#                              that cap on top of terminal-status detection.
#                              (default: unbounded)
#   POLL_INTERVAL_SECONDS   -- seconds between polls (default: 30)
#   MAX_CONSECUTIVE_HTTP_FAILURES
#                           -- API-health circuit breaker, independent of
#                              the test-plan timeout concept. Fails the
#                              build if Testsigma's API itself is
#                              unreachable this many polls in a row.
#                              (default: 10)
# -------------------------------------------------------------

set -uo pipefail
# NOTE: 'set -e' intentionally omitted. A single failed curl inside the
# polling loop must not kill the whole script -- every HTTP call is
# checked and handled explicitly instead.

RUN_ID_FILE="${RUN_ID_FILE:-/tmp/ts_run_id_manual-run}"
POLL_INTERVAL="${POLL_INTERVAL_SECONDS:-30}"
MAX_HTTP_FAILURES="${MAX_CONSECUTIVE_HTTP_FAILURES:-10}"

if [[ -z "${TS_API_TOKEN:-}" ]]; then
  echo "[ERROR] TS_API_TOKEN is not set."
  exit 1
fi

if [[ ! -f "$RUN_ID_FILE" ]]; then
  echo "[ERROR] Run ID file not found at ${RUN_ID_FILE}."
  echo "[ERROR] Did trigger-run.sh complete successfully?"
  exit 1
fi

RUN_ID=$(cat "$RUN_ID_FILE")

# --- Safety-cap normalization -------------------------------------------
RAW_MAX_WAIT="${MAX_WAIT_MINUTES:-0}"
RAW_MAX_WAIT_LC=$(echo "$RAW_MAX_WAIT" | tr '[:upper:]' '[:lower:]')

if [[ -z "$RAW_MAX_WAIT" || "$RAW_MAX_WAIT_LC" == "0" || "$RAW_MAX_WAIT_LC" == "unbounded" || "$RAW_MAX_WAIT_LC" == "none" ]]; then
  CAP_ENABLED=false
  MAX_SECONDS=0
  echo "[INFO] MAX_WAIT_MINUTES='${RAW_MAX_WAIT}' -> safety cap DISABLED. Polling until terminal status only."
else
  if ! [[ "$RAW_MAX_WAIT" =~ ^[0-9]+$ ]]; then
    echo "[ERROR] MAX_WAIT_MINUTES must be a non-negative integer, 'unbounded', or 'none'. Got: '${RAW_MAX_WAIT}'"
    exit 1
  fi
  CAP_ENABLED=true
  MAX_SECONDS=$(( RAW_MAX_WAIT * 60 ))
  echo "[INFO] MAX_WAIT_MINUTES='${RAW_MAX_WAIT}' -> safety cap ENABLED at ${MAX_SECONDS}s."
fi

ELAPSED=0
CONSECUTIVE_HTTP_FAILURES=0

echo "[INFO] Polling Testsigma for Run ID: ${RUN_ID}"

while true; do
  if [[ "$CAP_ENABLED" == true && "$ELAPSED" -ge "$MAX_SECONDS" ]]; then
    echo "[TIMEOUT] Safety cap of ${RAW_MAX_WAIT} minute(s) reached. Marking Jenkins build as FAILED."
    echo "[TIMEOUT] Testsigma Run ID ${RUN_ID} is left RUNNING in Testsigma -- this script does not stop it on timeout, by design."
    exit 1
  fi

  TMPFILE=$(mktemp)
  HTTP_CODE=$(curl -s -o "$TMPFILE" -w "%{http_code}" \
    -X GET \
    -H "Authorization: Bearer ${TS_API_TOKEN}" \
    -H "Accept: application/json" \
    "https://app.testsigma.com/api/v1/execution_results/${RUN_ID}")

  BODY=$(cat "$TMPFILE"); rm -f "$TMPFILE"

  if [[ "$HTTP_CODE" != "200" ]]; then
    CONSECUTIVE_HTTP_FAILURES=$(( CONSECUTIVE_HTTP_FAILURES + 1 ))
    echo "[WARN] Poll request failed with HTTP ${HTTP_CODE} (consecutive failure ${CONSECUTIVE_HTTP_FAILURES}/${MAX_HTTP_FAILURES})."
    echo "[WARN] Response body: ${BODY}"
    if [[ "$CONSECUTIVE_HTTP_FAILURES" -ge "$MAX_HTTP_FAILURES" ]]; then
      echo "[ERROR] Testsigma API unreachable for ${MAX_HTTP_FAILURES} consecutive polls. Failing build (API-health circuit breaker -- independent of the test-plan timeout)."
      exit 1
    fi
    sleep "$POLL_INTERVAL"; ELAPSED=$(( ELAPSED + POLL_INTERVAL )); continue
  fi

  CONSECUTIVE_HTTP_FAILURES=0

  RESULT=$(echo "$BODY" | python3 -c "import sys,json; print(json.load(sys.stdin).get('result','UNKNOWN'))" 2>/dev/null || echo "UNKNOWN")
  STATUS=$(echo "$BODY" | python3 -c "import sys,json; print(json.load(sys.stdin).get('status','UNKNOWN'))" 2>/dev/null || echo "UNKNOWN")

  echo "Run ID: ${RUN_ID} | result: ${RESULT} | status: ${STATUS} | elapsed: ${ELAPSED}s"

  case "$RESULT" in
    PASSED|SUCCESS|PASS)
      echo "[SUCCESS] Run completed with result: ${RESULT}."
      exit 0
      ;;
    FAILURE|FAILED|FAIL)
      echo "[FAILED] Run FAILED."
      exit 1
      ;;
    STOPPED|ABORTED|CANCELLED)
      echo "[STOPPED] Run stopped/aborted."
      exit 1
      ;;
    *)
      # Fail-safe: Testsigma's RESULT vocabulary has been observed to
      # differ from what was originally assumed here (this endpoint
      # returns "SUCCESS" for a pass, not "PASSED" -- confirmed against a
      # real run that otherwise polled forever). Rather than trust the
      # RESULT list above to be complete, also check STATUS: if Testsigma
      # itself considers the run finished (STATUS_COMPLETED) but RESULT
      # is a string we don't recognize, don't loop forever waiting for a
      # match that will never come -- fail loudly instead so it gets
      # noticed, and add the new RESULT value to the case above.
      if [[ "$STATUS" == STATUS_COMPLETED* ]]; then
        echo "[ERROR] STATUS='${STATUS}' looks terminal but RESULT='${RESULT}' isn't a recognized value (see the case statement in this script). Treating as a failure for safety rather than polling forever or assuming success. Add '${RESULT}' to the known RESULT list once confirmed."
        exit 1
      fi
      sleep "$POLL_INTERVAL"; ELAPSED=$(( ELAPSED + POLL_INTERVAL ))
      ;;
  esac
done

#!/bin/bash
# -------------------------------------------------------------
# fetch-report.sh
# Fetches JSON test results from Testsigma's v2 result-hierarchy
# API after a run completes. Which level(s) get fetched is chosen
# through the TS_REPORT_LEVELS environment variable, set once in
# the Jenkinsfile's environment{} block (see Part 7 of the guide) --
# no code edits required to change what gets pulled on the next run.
#
# Required environment variables (injected by Jenkins):
#   TS_API_TOKEN  -- Testsigma API key
#   RUN_ID_FILE   -- fixed path set in the Jenkinsfile environment block
#   REPORT_DIR    -- directory to write JSON output into
#
# Optional environment variables:
#   TS_REPORT_LEVELS        -- comma-separated subset of:
#                              PLAN, SUITE, CASE, STEP, ALL
#                              (default: SUITE,CASE,STEP)
#   TS_SUITE_RESULT_IDS     -- comma-separated suite_result_id values.
#                              OPTIONAL override -- auto-discovered if unset.
#   TS_CASE_RESULT_IDS      -- comma-separated case_result_id values.
#                              OPTIONAL override -- auto-discovered if unset.
#   REQUEST_PACING_SECONDS  -- delay between successive API calls in a
#                              loop, to reduce rate-limit risk (default: 1)
#
# AUTO-DISCOVERY:
# GET /test_runs/{run_id}/test_suite_results and
# GET /test_runs/{run_id}/test_case_results, called WITHOUT an id, return
# a paginated list of every suite/case result belonging to the run -- same
# content/totalPages shape as the per-case step_results endpoint. So
# SUITE and CASE don't require you to supply *_IDS at all: if the
# corresponding *_IDS var is unset, this script lists the run's actual
# suite/case results and uses those ids automatically. If you DO supply
# *_IDS, that's honoured as a manual override/filter -- useful if you only
# want a subset. TS_CASE_RESULT_IDS (or the auto-discovered case list) is
# also what STEP iterates over, so requesting STEP alone auto-discovers
# cases too.
#
# A MACHINE (per-device/browser) level existed in an earlier version of
# this script but has been removed entirely: Testsigma's API doesn't
# confirm support for listing machine results without a known id, so it
# never reached the same reliability as SUITE/CASE and added a parameter
# nobody used. If per-device breakdowns become useful later, that level
# can be reintroduced the same way SUITE/CASE work today.
#
# Design notes carried forward from earlier revisions:
#   - PLAN is always fetched, even if not explicitly requested, so the
#     report can never come back with zero files just because a deeper
#     level's ids were unavailable.
#   - Every level's outcome (OK/FAILED + why) is recorded to
#     <RUN_ID>_fetch-summary.jsonl in REPORT_DIR. generate-html-report.sh
#     reads this and renders it as a collapsed "Fetch Summary" section at
#     the top of every report.
# -------------------------------------------------------------

set -uo pipefail

if [[ -z "${TS_API_TOKEN:-}" ]]; then
  echo "[ERROR] TS_API_TOKEN is not set."
  exit 1
fi

RUN_ID_FILE="${RUN_ID_FILE:-/tmp/ts_run_id_manual-run}"
if [[ ! -f "$RUN_ID_FILE" ]]; then
  echo "[ERROR] Run ID file not found at ${RUN_ID_FILE}. Did trigger-run.sh complete successfully?"
  exit 1
fi
RUN_ID=$(cat "$RUN_ID_FILE")

REPORT_DIR="${REPORT_DIR:-./ts-reports}"
mkdir -p "$REPORT_DIR" || { echo "[ERROR] Could not create REPORT_DIR at ${REPORT_DIR}."; exit 1; }

LEVELS_RAW="${TS_REPORT_LEVELS:-SUITE,CASE,STEP}"
PACING="${REQUEST_PACING_SECONDS:-1}"

IFS=',' read -ra LEVELS <<< "$(echo "$LEVELS_RAW" | tr '[:lower:]' '[:upper:]' | tr -d ' ')"
if [[ " ${LEVELS[*]} " == *" ALL "* ]]; then
  LEVELS=(PLAN SUITE CASE STEP)
fi

echo "[INFO] Run ID: ${RUN_ID} | Requested levels: ${LEVELS[*]}"

# --- Fetch-summary manifest -----------------------------------------------
SUMMARY_FILE="${REPORT_DIR}/${RUN_ID}_fetch-summary.jsonl"
: > "$SUMMARY_FILE"

record_summary() {
  python3 -c "
import json, sys
level, status, detail, summary_file = sys.argv[1], sys.argv[2], sys.argv[3], sys.argv[4]
with open(summary_file, 'a') as f:
    f.write(json.dumps({'level': level, 'status': status, 'detail': detail}) + chr(10))
" "$1" "$2" "$3" "$SUMMARY_FILE"
}

# --- Always fetch PLAN as a baseline, even if not explicitly requested ----
PLAN_EXPLICIT=false
if [[ " ${LEVELS[*]} " == *" PLAN "* ]]; then
  PLAN_EXPLICIT=true
else
  echo "[INFO] PLAN was not explicitly requested -- fetching it anyway as a baseline so the report is never completely empty."
  LEVELS+=(PLAN)
fi

# --- Shared GET helper with retry + JSON validation ----------------------
ts_get() {
  local url="$1"
  local outfile="$2"
  local attempt=0
  local max_attempts=3

  while (( attempt < max_attempts )); do
    attempt=$(( attempt + 1 ))
    local tmpfile
    tmpfile=$(mktemp)
    local http_code
    http_code=$(curl -s -o "$tmpfile" -w "%{http_code}" \
      -X GET \
      -H "Authorization: Bearer ${TS_API_TOKEN}" \
      -H "Accept: application/json" \
      "$url")

    if [[ "$http_code" == "200" ]]; then
      if python3 -c "import json; json.load(open('$tmpfile'))" 2>/dev/null; then
        mv "$tmpfile" "$outfile"
        return 0
      else
        echo "[ERROR] Response from ${url} was HTTP 200 but not valid JSON. Body saved to ${outfile}.invalid"
        mv "$tmpfile" "${outfile}.invalid"
        return 1
      fi
    fi

    # 404 on a *list* endpoint (no id) can legitimately mean "this level has
    # no list-without-id form" rather than a transient failure -- don't
    # burn all 3 retry attempts on a 404, fail fast so the caller can decide
    # whether to fall back.
    if [[ "$http_code" == "404" ]]; then
      rm -f "$tmpfile"
      return 2
    fi

    echo "[WARN] GET ${url} returned HTTP ${http_code} (attempt ${attempt}/${max_attempts})."
    cat "$tmpfile"
    rm -f "$tmpfile"
    sleep "$PACING"
  done

  echo "[ERROR] GET ${url} failed after ${max_attempts} attempts."
  return 1
}

# --- Generic paginated-list walker ----------------------------------------
# Walks a list-without-id endpoint (content/totalPages shape), writing one
# JSON file per item (named <out_prefix><id>.json) and appending each
# item's id to id_out_file (one per line). Returns 0 on success (even zero
# items), 2 if the endpoint doesn't support the list form at all (404 on
# the very first page -- treated as "not available", not an error), 1 on
# any other failure.
ts_get_paginated_list() {
  local base_url="$1"
  local out_prefix="$2"
  local id_out_file="$3"
  local label="$4"
  local page=0
  local total_pages=1
  : > "$id_out_file"
  while (( page < total_pages )); do
    local paged_file
    paged_file=$(mktemp)
    ts_get "${base_url}?page=${page}&size=100" "$paged_file"
    local get_status=$?
    if [[ $get_status -eq 2 && $page -eq 0 ]]; then
      rm -f "$paged_file"
      return 2
    elif [[ $get_status -ne 0 ]]; then
      rm -f "$paged_file"
      return 1
    fi
    local flag_file
    flag_file=$(mktemp)
    python3 -c "
import json
with open('$paged_file') as f:
    data = json.load(f)
content = data.get('content', [])
for item in content:
    iid = item.get('id')
    if iid is not None:
        with open('${out_prefix}' + str(iid) + '.json', 'w') as out:
            json.dump(item, out)
        with open('$id_out_file', 'a') as idf:
            idf.write(str(iid) + chr(10))
total_pages = data.get('totalPages', 1) or 1
with open('$flag_file', 'w') as flag:
    flag.write(str(total_pages) + '|' + str(len(content)))
" || { echo "[ERROR] Failed to parse ${label} list response (page ${page})."; rm -f "$paged_file" "$flag_file"; return 1; }
    local flag_contents
    flag_contents=$(cat "$flag_file")
    total_pages="${flag_contents%%|*}"
    local page_count="${flag_contents##*|}"
    rm -f "$paged_file" "$flag_file"
    echo "[INFO]   ${label} list page $((page + 1))/${total_pages} -- ${page_count} item(s)"
    if [[ "$page_count" == "0" && "$page" -gt 0 ]]; then
      echo "[WARN] ${label} list page ${page} returned 0 items but totalPages=${total_pages} -- stopping early."
      break
    fi
    page=$(( page + 1 ))
    sleep "$PACING"
  done
  return 0
}

FAILURES=0

# --- PLAN level -----------------------------------------------------------
if [[ " ${LEVELS[*]} " == *" PLAN "* ]]; then
  echo "[INFO] Fetching PLAN level..."
  if ts_get "https://app.testsigma.com/api/v2/test_runs/${RUN_ID}" "${REPORT_DIR}/${RUN_ID}_plan.json"; then
    echo "[OK] PLAN level -> ${REPORT_DIR}/${RUN_ID}_plan.json"
    if [[ "$PLAN_EXPLICIT" == "true" ]]; then
      record_summary "PLAN" "OK" "${RUN_ID}_plan.json"
    else
      record_summary "PLAN" "OK (baseline)" "Not explicitly requested; fetched anyway as a baseline -> ${RUN_ID}_plan.json"
    fi
  else
    FAILURES=$(( FAILURES + 1 ))
    record_summary "PLAN" "FAILED" "API request failed -- see console log above"
  fi
  sleep "$PACING"
fi

# --- SUITE level (auto-discovered unless TS_SUITE_RESULT_IDS overrides) ---
if [[ " ${LEVELS[*]} " == *" SUITE "* ]]; then
  if [[ -n "${TS_SUITE_RESULT_IDS:-}" ]]; then
    IFS=',' read -ra SUITE_IDS <<< "$(echo "$TS_SUITE_RESULT_IDS" | tr -d ' ')"
    S_OK=0; S_FAIL=0
    if [[ ${#SUITE_IDS[@]} -gt 0 ]]; then
      for sid in "${SUITE_IDS[@]}"; do
        echo "[INFO] Fetching SUITE level for suite_result_id=${sid} (manual override)..."
        if ts_get "https://app.testsigma.com/api/v2/test_runs/${RUN_ID}/test_suite_results/${sid}" "${REPORT_DIR}/${RUN_ID}_suite_${sid}.json"; then
          echo "[OK] SUITE ${sid} -> ${REPORT_DIR}/${RUN_ID}_suite_${sid}.json"
          S_OK=$(( S_OK + 1 ))
        else
          FAILURES=$(( FAILURES + 1 )); S_FAIL=$(( S_FAIL + 1 ))
        fi
        sleep "$PACING"
      done

    else
      record_summary "SUITE" "OK" "${S_OK}/${#SUITE_IDS[@]} manually-supplied suite id(s) fetched"
    fi

# --- Resolve case ids once, shared by CASE and STEP ------------------------
CASE_IDS_RESOLVED=()
CASE_IDS_SOURCE=""   # "manual" or "auto" or "failed"
resolve_case_ids() {
  if [[ ${#CASE_IDS_RESOLVED[@]} -gt 0 || "$CASE_IDS_SOURCE" == "failed" ]]; then
    return
  fi
  if [[ -n "${TS_CASE_RESULT_IDS:-}" ]]; then
    IFS=',' read -ra CASE_IDS_RESOLVED <<< "$(echo "$TS_CASE_RESULT_IDS" | tr -d ' ')"
    CASE_IDS_SOURCE="manual"
  else
    echo "[INFO] Auto-discovering case ids for this run (no TS_CASE_RESULT_IDS supplied)..."
    CASE_ID_FILE=$(mktemp)
    ts_get_paginated_list "https://app.testsigma.com/api/v2/test_runs/${RUN_ID}/test_case_results" "${REPORT_DIR}/${RUN_ID}_case_" "$CASE_ID_FILE" "CASE"
    if [[ $? -eq 0 ]]; then
      # NOTE: deliberately NOT using `mapfile`/`readarray` here -- both are
      # bash 4+ only, and macOS ships bash 3.2 as /bin/bash permanently (Apple
      # hasn't updated it since 2007, for GPLv3 licensing reasons) regardless
      # of what newer bash a user may have installed separately. `mapfile`
      # fails with "command not found" on every Mac agent using the system
      # bash, which this script's shebang resolves to. This loop works on
      # bash 3.2 and every later version.
      CASE_IDS_RESOLVED=()
      while IFS= read -r line; do
        [[ -n "$line" ]] && CASE_IDS_RESOLVED+=("$line")
      done < "$CASE_ID_FILE"
      CASE_IDS_SOURCE="auto"
    else
      CASE_IDS_SOURCE="failed"
    fi
    rm -f "$CASE_ID_FILE"
  fi
}

      TOTAL_PAGES=1
      CASE_FAILED=false

        PAGED_FILE=$(mktemp)
        if ts_get "https://app.testsigma.com/api/v2/test_runs/${RUN_ID}/test_case_results/${cid}/step_results?page=${PAGE}&size=100" "$PAGED_FILE"; then

          echo "[INFO]   page $((PAGE + 1))/${TOTAL_PAGES} -- ${PAGE_COUNT} step(s)"
          if [[ "$PAGE_COUNT" == "0" && "$PAGE" -gt 0 ]]; then
            echo "[WARN] Page ${PAGE} returned 0 steps but totalPages=${TOTAL_PAGES} -- stopping early."
            break
          fi
          PAGE=$(( PAGE + 1 ))
          sleep "$PACING"
        else
          FAILURES=$(( FAILURES + 1 ))
          CASE_FAILED=true
          rm -f "$PAGED_FILE"
          break
        fi
      done
      TOTAL_STEPS=$(wc -l < "$OUT_JSONL" | tr -d ' ')
      ST_TOTAL_STEPS=$(( ST_TOTAL_STEPS + TOTAL_STEPS ))
      if [[ "$CASE_FAILED" == "true" ]]; then
        ST_FAIL=$(( ST_FAIL + 1 ))
      else
        ST_OK=$(( ST_OK + 1 ))
      fi
      echo "[OK] STEP results for case ${cid} -> ${OUT_JSONL} (${TOTAL_PAGES} page(s), ${TOTAL_STEPS} step(s) total)"
    done
    fi
    src_label="${CASE_IDS_SOURCE}-discovered"
    if [[ "$CASE_IDS_SOURCE" == "manual" ]]; then src_label="manually-supplied"; fi
    if (( ST_FAIL > 0 )); then
      record_summary "STEP" "PARTIAL" "${ST_OK} case(s) ok, ${ST_FAIL} case(s) failed, ${ST_TOTAL_STEPS} step(s) fetched total (${src_label} case ids)"
    else
      record_summary "STEP" "OK" "${ST_OK} case(s), ${ST_TOTAL_STEPS} step(s) fetched total (${src_label} case ids)"
    fi
  fi
fi

echo "[INFO] Fetch complete. Output directory: ${REPORT_DIR}"

if (( FAILURES > 0 )); then
  echo "[ERROR] ${FAILURES} fetch operation(s) failed. See log above."
  exit 1
fi

exit 0

def load_json(path):
    with open(path) as f:
        return json.load(f)

def load_jsonl(path):
    items = []
    with open(path) as f:
        for line in f:
            line = line.strip()
            if line:
                items.append(json.loads(line))
    return items

def esc(v):
    if v is None:
        return ""
    return html_lib.escape(str(v))

def flatten(obj, prefix=""):
    out = {}
    if isinstance(obj, dict):
        for k, v in obj.items():
            key = "{}.{}".format(prefix, k) if prefix else k
            if isinstance(v, dict):
                out.update(flatten(v, key))
            else:
                out[key] = v
    return out

def fmt_duration(v):
    # Testsigma reports duration in milliseconds. Raw "156000" reads as
    # noise; render as seconds/minutes instead. Returns None (caller falls
    # back to the raw value) if v isn't a plain number.
    try:
        ms = float(v)
    except (TypeError, ValueError):
        return None
    total_seconds = ms / 1000.0
    if total_seconds < 60:
        return "{:.1f}s".format(total_seconds)
    minutes = int(total_seconds // 60)
    seconds = int(total_seconds % 60)
    return "{}m {}s".format(minutes, seconds)

# Curated order for the Test Plan Summary -- answers "what happened" at a
# glance instead of reading like a raw API dump. Everything not listed
# here still appears, just tucked into a collapsed <details> block below
# (native HTML, no JS needed) so the report reads as a summary first and
# a full technical dump second, rather than the other way around.
PLAN_SUMMARY_PRIORITY = [
    "metrics.result", "metrics.duration", "startTime", "endTime",
    "triggeredType", "buildNo",
    "metrics.totalCount", "metrics.passedCount", "metrics.failedCount",
    "metrics.stoppedCount", "metrics.notExecutedCount",
]

def render_plan_summary(obj, table_id):
    flat = flatten(obj)

    def row(k, v):
        if k.lower().endswith("duration") and v not in (None, ""):
            formatted = fmt_duration(v)
            v = formatted if formatted is not None else v
        return "<tr><th>{}</th><td>{}</td></tr>".format(esc(pretty_label(k)), esc(v))

    primary_rows = "".join(row(k, flat[k]) for k in PLAN_SUMMARY_PRIORITY if k in flat)

    # Everything else -- internal IDs, executionDetails.* config, secondary
    # consolidated-count fields, etc. Kept, not deleted; just not the first
    # thing a reader has to wade through. childRuns is dropped entirely
    # when empty (the overwhelmingly common case for a single, non-rerun
    # plan) since "childRuns: []" tells a reader nothing.
    shown = set(PLAN_SUMMARY_PRIORITY)
    remaining = {k: v for k, v in flat.items() if k not in shown and not (k == "childRuns" and v in ("[]", [], None))}
    extra_rows = "".join(row(k, v) for k, v in remaining.items())
    details_block = ""
    if extra_rows:
        details_block = (
            '<details class="plan-extra"><summary>Additional execution details '
            '({n} more field{s})</summary><table class="kv"><tbody>{rows}</tbody></table></details>'
        ).format(n=len(remaining), s="" if len(remaining) == 1 else "s", rows=extra_rows)

    return (
        '<section><h2>Test Plan Summary</h2><table class="kv"><tbody>{primary}</tbody></table>{details}</section>'
    ).format(primary=primary_rows, details=details_block)

def render_table(title, rows, columns, table_id):
    n = len(rows)
    paginate = n > 20
    thead = "".join("<th>{}</th>".format(esc(pretty_label(c))) for c in columns)
    body_rows = []
    for r in rows:
        cells = []
        for c in columns:
            val = r.get(c, "")
            if c.lower().endswith("duration") and val not in (None, ""):
                formatted = fmt_duration(val)
                cells.append("<td>{}</td>".format(esc(formatted if formatted is not None else val)))
            else:
                cells.append("<td>{}</td>".format(esc(val)))
        body_rows.append("<tr>{}</tr>".format("".join(cells)))
    tbody = "".join(body_rows)
    pager = ""
    if paginate:
        pager = (
            '<div class="pager">'
            '<button id="{tid}_prev">&laquo; Prev</button>'
            '<span id="{tid}_pageLabel"></span>'
            '<button id="{tid}_next">Next &raquo;</button>'
            '</div>'
        ).format(tid=table_id)
        table_scripts.append(table_id)
    return (
        '<section><h2>{title} <span class="count">({n} entries)</span></h2>'
        '<table id="{tid}" class="data"><thead><tr>{thead}</tr></thead>'
        '<tbody>{tbody}</tbody></table>{pager}</section>'
    ).format(title=esc(title), n=n, tid=table_id, thead=thead, tbody=tbody, pager=pager)

# Steps are grouped by testCaseId (the logical test case), not by
# caseResultId (a single execution instance of that case) -- so two
# execution rows for the same case (e.g. a retry) land in one group.
# Each test case is rendered as its own PAGE (one testCaseId visible at a
# time, Prev/Next to move between them) -- not stacked one after another.
# Within a page, if that test case has more than STEP_FOLD_THRESHOLD
# steps, its table is collapsed into a single <details> toggle with no
# inner pagination; at or under the threshold it renders open.
STEP_FOLD_THRESHOLD = 60

# caseResultId, stepId, type, action, message, result, errorType, duration
# -- percentages sum to 100. table-layout:fixed (set on .step-data in the
# stylesheet) makes the browser honor these widths regardless of content,
# and word-break/overflow-wrap on the cells (also in the stylesheet) wraps
# long, unbroken text (locators, URLs) inside its column instead of
# growing the column -- this is what was pushing Error Type/Duration out
# of the report's bounds: table-layout:auto (the default) sizes columns
# to fit their widest single unbroken token with no wrap, so one long
# Message/Action value stretched the whole table past its container.
STEP_COLUMNS = ["caseResultId", "stepId", "type", "action", "message", "result", "errorType", "duration"]
STEP_COLUMN_WIDTHS = [11, 7, 9, 19, 20, 13, 14, 7]
# Only these two columns hold genuinely unbounded free text (locators,
# URLs, arbitrary messages) and need aggressive word-break/overflow-wrap.
# Everything else (IDs, Type, Result, Error Type, Duration) is a short,
# bounded vocabulary (SUCCESS/FAILURE/NOT_EXECUTED, NLP_TEXT, "21.8s", ...)
# that should never be force-broken mid-word -- those columns instead get
# single-line + ellipsis (with the full value in a title tooltip) as a
# defensive fallback, so a value like "SUCCESS" never gets mangled into
# "SUCCES"/"S" the way a blanket overflow-wrap:anywhere would do.
STEP_WRAP_COLUMNS = {"action", "message"}

def render_step_rows_table(rows):
    colgroup = "".join('<col style="width:{}%">'.format(w) for w in STEP_COLUMN_WIDTHS)
    thead_cells = []
    for c in STEP_COLUMNS:
        label = esc(pretty_label(c))
        thead_cells.append('<th title="{label}">{label}</th>'.format(label=label))
    thead = "".join(thead_cells)
    body_rows = []
    for r in rows:
        cells = []
        for c in STEP_COLUMNS:
            val = r.get(c, "")
            if c.lower().endswith("duration") and val not in (None, ""):
                formatted = fmt_duration(val)
                val = formatted if formatted is not None else val
            esc_val = esc(val)
            if c in STEP_WRAP_COLUMNS:
                cells.append("<td>{}</td>".format(esc_val))
            else:
                cells.append('<td title="{v}">{v}</td>'.format(v=esc_val))
        body_rows.append("<tr>{}</tr>".format("".join(cells)))
    return (
        '<table class="data step-data"><colgroup>{colgroup}</colgroup>'
        '<thead><tr>{thead}</tr></thead><tbody>{tbody}</tbody></table>'
    ).format(colgroup=colgroup, thead=thead, tbody="".join(body_rows))

def render_step_groups(steps, title, group_pager_scripts):
    n_total = len(steps)
    groups = {}
    order = []
    for s in steps:
        tc_id = s.get("testCaseId", "")
        if tc_id not in groups:
            groups[tc_id] = []
            order.append(tc_id)
        groups[tc_id].append(s)

# poll-status.ps1

#                             terminal-status detection. (default: unbounded)

        $elapsed += $pollInterval
        continue
    }

    $result = if ($response.result) { $response.result } else { "UNKNOWN" }
    $status = if ($response.status) { $response.status } else { "UNKNOWN" }

            Write-Host "[SUCCESS] Run completed with result: $result."
            exit 0
        }
        "^(FAILURE|FAILED|FAIL)$" {
            Write-Host "[FAILED] Run FAILED."
            exit 1
        }
        "^(STOPPED|ABORTED|CANCELLED)$" {
            Write-Host "[STOPPED] Run stopped/aborted."
            exit 1

        default {
            # Fail-safe: Testsigma's result vocabulary has been observed to
            # differ from what was originally assumed here (this endpoint
            # returns "SUCCESS" for a pass, not "PASSED" -- confirmed
            # against a real run that otherwise polled forever). Rather
            # than trust the list above to be complete, also check status:
            # if Testsigma itself considers the run finished
            # (STATUS_COMPLETED) but result is a string not recognized
            # above, don't loop forever waiting for a match that will
            # never come -- fail loudly instead so it gets noticed, and
            # add the new result value to the patterns above.
            if ($status -like "STATUS_COMPLETED*") {
                Write-Host "[ERROR] status='$status' looks terminal but result='$result' isn't a recognized value (see the switch statement in this script). Treating as a failure for safety rather than polling forever or assuming success. Add '$result' to the known result list once confirmed."
                exit 1
            }
            Start-Sleep -Seconds $pollInterval
            $elapsed += $pollInterval
        }
    }

# Walks a list-without-id endpoint, writing one JSON file per item
# (named <OutPrefix><id>.json). Returns a hashtable: @{ Status = 0/1/2; Ids = @(...) }
function Invoke-TsGetPaginatedList {
    param([string]$BaseUrl, [string]$OutPrefix, [string]$Label)
    $page = 0
    $totalPages = 1
    $ids = New-Object System.Collections.Generic.List[string]
    while ($page -lt $totalPages) {
        try {
            $resp = Invoke-RestMethod -Uri "$BaseUrl`?page=$page&size=100" -Method Get `
                -Headers @{ "Authorization" = "Bearer $($env:TS_API_TOKEN)"; "Accept" = "application/json" }
        }
        catch {
            $statusCode = $null
            if ($_.Exception.Response) { $statusCode = [int]$_.Exception.Response.StatusCode }
            if ($statusCode -eq 404 -and $page -eq 0) { return @{ Status = 2; Ids = @() } }
            Write-Host "[ERROR] ${Label} list request failed (page $page): $($_.Exception.Message)"
            return @{ Status = 1; Ids = @() }
        }
        $pageCount = 0
        foreach ($item in $resp.content) {
            if ($null -ne $item.id) {
                ($item | ConvertTo-Json -Depth 20) | Set-Content -Path "$OutPrefix$($item.id).json"
                $ids.Add([string]$item.id)
                $pageCount++
            }
        }
        $totalPages = if ($resp.totalPages) { [int]$resp.totalPages } else { 1 }
        Write-Host "[INFO]   $Label list page $($page + 1)/$totalPages -- $pageCount item(s)"
        if ($pageCount -eq 0 -and $page -gt 0) {
            Write-Host "[WARN] $Label list page $page returned 0 items but totalPages=$totalPages -- stopping early."
            break
        }
        $page++
        Start-Sleep -Seconds $pacing
    }
    return @{ Status = 0; Ids = $ids }

    function Render-PlanSummary($obj, [string]$tableId) {
        $flat = Flatten-Object -obj $obj

        $rowFn = {
            param($k, $v)
            if ($k.ToLower().EndsWith("duration") -and $null -ne $v -and $v -ne "") {
                $formatted = Format-Duration $v
                if ($null -ne $formatted) { $v = $formatted }
            }
            "<tr><th>$(HtmlEncode (Get-PrettyLabel $k))</th><td>$(HtmlEncode $v)</td></tr>"
        }

        $primaryRows = ($PlanSummaryPriority | Where-Object { $flat.Contains($_) } | ForEach-Object { & $rowFn $_ $flat[$_] }) -join ""

        $shown = [System.Collections.Generic.HashSet[string]]::new([string[]]$PlanSummaryPriority)
        $remainingKeys = $flat.Keys | Where-Object {
            (-not $shown.Contains($_)) -and (-not ($_ -eq "childRuns" -and ($flat[$_] -eq "[]" -or $null -eq $flat[$_] -or ($flat[$_] -is [array] -and $flat[$_].Count -eq 0))))
        }
        $extraRows = ($remainingKeys | ForEach-Object { & $rowFn $_ $flat[$_] }) -join ""

        $detailsBlock = ""
        if ($extraRows) {
            $n = ($remainingKeys | Measure-Object).Count
            $plural = if ($n -eq 1) { "" } else { "s" }
            $detailsBlock = "<details class=""plan-extra""><summary>Additional execution details ($n more field$plural)</summary><table class=""kv""><tbody>$extraRows</tbody></table></details>"
        }

        return "<section><h2>Test Plan Summary</h2><table class=""kv""><tbody>$primaryRows</tbody></table>$detailsBlock</section>"
    }



    function Render-Table([string]$title, [array]$rows, [array]$columns, [string]$tableId) {
        $n = $rows.Count
        $paginate = $n -gt 20
        $thead = ($columns | ForEach-Object { "<th>$(HtmlEncode (Get-PrettyLabel $_))</th>" }) -join ""
        $bodyRows = foreach ($r in $rows) {
            $cells = ($columns | ForEach-Object {
                $val = if ($r.Contains($_)) { $r[$_] } else { "" }
                if ($_.ToLower().EndsWith("duration") -and $val -ne $null -and $val -ne "") {
                    $formatted = Format-Duration $val
                    if ($null -ne $formatted) { $val = $formatted }
                }
                "<td>$(HtmlEncode $val)</td>"
            }) -join ""
            "<tr>$cells</tr>"
        }
        $tbody = ($bodyRows -join "")
        $pager = ""
        if ($paginate) {
            $pager = "<div class=""pager""><button id=""$($tableId)_prev"">&laquo; Prev</button><span id=""$($tableId)_pageLabel""></span><button id=""$($tableId)_next"">Next &raquo;</button></div>"
            $tableScripts.Add($tableId)
        }
        return "<section><h2>$(HtmlEncode $title) <span class=""count"">($n entries)</span></h2><table id=""$tableId"" class=""data""><thead><tr>$thead</tr></thead><tbody>$tbody</tbody></table>$pager</section>"
    }

    # Steps are grouped by testCaseId (the logical test case), not
    # caseResultId (one execution instance of it) -- so a retry that
    # produces a second caseResultId for the same case lands in the same
    # group. Each test case renders as its own PAGE (one visible at a
    # time, Prev/Next to move between them) rather than stacked one after
    # another. Within a page, a test case with more than
    # StepFoldThreshold steps collapses into a single <details> toggle
    # with no inner pagination; at or under the threshold it renders open.
    # Mirrors generate-html-report.sh.
    $StepFoldThreshold = 60

    # caseResultId, stepId, type, action, message, result, errorType,
    # duration -- percentages sum to 100, measured against the report's
    # actual font stack/size so common values (SUCCESS, NOT_EXECUTED,
    # ASSERT_ERROR, "Case Result ID" header) fit on one line without
    # truncating. table-layout:fixed (set on .step-data in the stylesheet)
    # makes the browser honor these widths regardless of content, which
    # is what fixes the original overflow bug: table-layout:auto (the
    # default) sizes a column to fit its single widest unbroken token with
    # no wrap, so one long Message/Action locator/URL stretched the whole
    # table past the report's bounds and pushed Error Type/Duration out of
    # view. Only Action/Message (genuinely unbounded free text) get
    # aggressive word-break in the stylesheet; every other column here is
    # a short, bounded vocabulary and stays single-line with an
    # ellipsis+title fallback instead of being force-broken mid-word.
    $StepColumns = @("caseResultId","stepId","type","action","message","result","errorType","duration")
    $StepColumnWidths = @(11, 7, 9, 19, 20, 13, 14, 7)

    function Render-StepRowsTable([array]$rows) {
        $colgroup = ($StepColumnWidths | ForEach-Object { "<col style=""width:$($_)%"">" }) -join ""
        $thead = ($StepColumns | ForEach-Object {
            $label = HtmlEncode (Get-PrettyLabel $_)
            "<th title=""$label"">$label</th>"
        }) -join ""
        $bodyRows = foreach ($r in $rows) {
            $cells = ($StepColumns | ForEach-Object {
                $val = if ($r.Contains($_)) { $r[$_] } else { "" }
                if ($_.ToLower().EndsWith("duration") -and $null -ne $val -and $val -ne "") {
                    $formatted = Format-Duration $val
                    if ($null -ne $formatted) { $val = $formatted }
                }
                $encoded = HtmlEncode $val
                if ($_ -eq "action" -or $_ -eq "message") {
                    "<td>$encoded</td>"
                } else {
                    "<td title=""$encoded"">$encoded</td>"
                }
            }) -join ""
            "<tr>$cells</tr>"
        }
        return "<table class=""data step-data""><colgroup>$colgroup</colgroup><thead><tr>$thead</tr></thead><tbody>$($bodyRows -join '')</tbody></table>"
    }

    function Render-StepGroups([array]$steps, [string]$title, [System.Collections.Generic.List[string]]$groupPagerScripts) {
        $nTotal = $steps.Count
        $groups = [ordered]@{}
        foreach ($s in $steps) {
            $tcId = [string]$s["testCaseId"]
            if (-not $groups.Contains($tcId)) { $groups[$tcId] = New-Object System.Collections.Generic.List[hashtable] }
            $groups[$tcId].Add($s)
        }

        # Group order = execution order (insertion order of $groups, which
        # follows $stepFiles in glob-sorted/caseResultId order) -- not a
        # re-sort by testCaseId value. Keeps the report readable as "what
        # happened, in the order it happened."
        $pageHtmlParts = foreach ($tcId in $groups.Keys) {
            $rows = $groups[$tcId]
            $n = $rows.Count
            $execIds = ($rows | ForEach-Object { [string]$_["caseResultId"] } | Select-Object -Unique | Sort-Object)
            $execLabel = $execIds -join ", "
            $hasFailure = $false
            foreach ($r in $rows) { if ([string]$r["result"] -eq "FAILURE") { $hasFailure = $true; break } }
            $badge = if ($hasFailure) { '<span class="sum-badge sum-failed">has failures</span>' } else { '<span class="sum-badge sum-ok">no failures</span>' }
            $header = "<div class=""step-case-header""><span class=""step-case-title"">Test Case $(HtmlEncode $tcId)</span> <span class=""count"">($n step(s) across execution(s): $(HtmlEncode $execLabel))</span> $badge</div>"
            $table = Render-StepRowsTable $rows

            $content = if ($n -gt $StepFoldThreshold) {
                $summaryText = "Test Case $(HtmlEncode $tcId) -- $n steps, folded (click to expand)"
                "<details class=""step-case-fold""><summary>$summaryText</summary>$header$table</details>"
            } else {
                "<div class=""step-case"">$header$table</div>"
            }
            "<div class=""step-case-page"">$content</div>"
        }

        $g = $groups.Keys.Count
        $pager = ""
        if ($g -gt 1) {
            $pager = '<div class="pager"><button id="stepPager_prev">&laquo; Prev</button><span id="stepPager_pageLabel"></span><button id="stepPager_next">Next &raquo;</button></div>'
            $groupPagerScripts.Add("stepPager")
        }

        return "<section><h2>$(HtmlEncode $title) <span class=""count"">($nTotal entries across $g test case(s))</span></h2><div id=""stepGroupsContainer"">$($pageHtmlParts -join '')</div>$pager</section>"
    }

    function Render-FetchSummary([array]$entries) {
        if (-not $entries -or $entries.Count -eq 0) { return "" }
        $statusClass = @{ "OK" = "sum-ok"; "OK (baseline)" = "sum-baseline"; "PARTIAL" = "sum-partial"; "FAILED" = "sum-failed" }
        $issueCount = ($entries | Where-Object { $_.status -eq "FAILED" -or $_.status -eq "PARTIAL" } | Measure-Object).Count
        $rows = ($entries | ForEach-Object {
            $level = HtmlEncode $_.level
            $status = $_.status
            $detail = HtmlEncode $_.detail
            $cls = if ($statusClass.ContainsKey($status)) { $statusClass[$status] } else { "sum-partial" }
            "<tr><td class=""sum-level"">$level</td><td><span class=""sum-badge $cls"">$(HtmlEncode $status)</span></td><td class=""sum-detail"">$detail</td></tr>"
        }) -join ""
        # Collapsed by default -- most runs fetch cleanly and this is a detail
        # panel, not the headline. The toggle label itself still surfaces
        # whether anything needs attention, so a problem is never fully hidden.
        if ($issueCount -gt 0) {
            $plural = if ($issueCount -eq 1) { "" } else { "s" }
            $verb = if ($issueCount -eq 1) { "needs" } else { "need" }
            $toggleLabel = "Fetch Summary -- $issueCount level$plural $verb attention (click to expand)"
        } else {
            $toggleLabel = "Fetch Summary -- everything fetched as expected (click to expand)"
        }
        return "<details class=""fetch-summary""><summary>$toggleLabel</summary><p class=""sub"">What was requested for this run, and what actually happened.</p><table class=""summary""><thead><tr><th>Level</th><th>Status</th><th>Detail</th></tr></thead><tbody>$rows</tbody></table></details>"
    }

    $suiteFiles = Get-ChildItem -Path $reportDir -Filter "$($runId)_suite_*.json" -ErrorAction SilentlyContinue | Sort-Object Name

        # Same nesting as SUITE -- confirmed against a real test_case_results response.
        $cols = @("id","testCaseId","metrics.result","metrics.duration","metrics.totalCount","metrics.passedCount","metrics.failedCount","metrics.notExecutedCount")
        [void]$sections.Append((Render-Table "Test Case Results" $rows $cols "caseTable"))
    }

    $stepFiles = Get-ChildItem -Path $reportDir -Filter "$($runId)_steps_*.jsonl" -ErrorAction SilentlyContinue | Sort-Object Name
    if ($stepFiles) {
        # Map caseResultId -> testCaseId using the case-level files loaded
        # above for Test Case Results. testCaseId is the logical test case;
        # caseResultId is one execution instance of it (a retry produces a
        # second caseResultId for the same testCaseId). Steps only carry
        # caseResultId natively, so this map is required to group by the
        # logical case. Falls back to grouping by caseResultId itself if a
        # given case file can't be read/parsed, so steps are never silently
        # dropped from the report.
        $caseResultToTestCase = @{}
        if ($caseFiles) {
            foreach ($cf in $caseFiles) {
                try {
                    $c = Flatten-Object -obj (Get-Content $cf.FullName -Raw | ConvertFrom-Json)
                    $crId = [string]$c["id"]
                    if ($crId) {
                        $tcId = if ($c.Contains("testCaseId")) { [string]$c["testCaseId"] } else { $crId }
                        $caseResultToTestCase[$crId] = $tcId
                    }
                } catch {
                    # Write-Host, not stdout redirection -- this script has no
                    # OUT_PATH-style stdout capture, so this is safe to log
                    # directly rather than silently swallow.
                    Write-Host "[WARN] Could not read/parse $($cf.FullName) while building the caseResultId->testCaseId map ($($_.Exception.Message)); its step(s) will fall back to grouping by caseResultId instead of testCaseId."
                }
            }
        }

        $allSteps = New-Object System.Collections.Generic.List[hashtable]
        foreach ($sf in $stepFiles) {
            $caseId = $sf.BaseName -replace '.*_steps_', ''
            $testCaseId = if ($caseResultToTestCase.ContainsKey($caseId)) { $caseResultToTestCase[$caseId] } else { $caseId }
            Get-Content $sf.FullName | ForEach-Object {
                if ($_.Trim()) {
                    $obj = $_ | ConvertFrom-Json
                    $flat = Flatten-Object -obj $obj
                    $flat["caseResultId"] = $caseId
                    $flat["testCaseId"] = $testCaseId
                    $allSteps.Add($flat)
                }
            }
        }
        [void]$sections.Append((Render-StepGroups $allSteps.ToArray() "Test Step Results" $groupPagerScripts))
    }

    # fetch-report.ps1 this should now be rare, but kept as a last-resort message.
    if (-not (Test-Path $planPath) -and -not ($suiteFiles -or $caseFiles -or $stepFiles)) {

    }

Part 10 — Choosing Report Levels & the 4-Hour Failsafe

The failsafe (MAX_WAIT_MINUTES). The job now waits primarily for the TestPlan to leave RUNNING — it does not just count down a fixed timer the way the original guide did. The 4-hour default only exists to catch a genuinely stuck plan; it is a safety net, not a target duration.

Value you enterBehavior
240 (default)Poll on terminal status; if nothing happens for 4 hours, fail the Jenkins build. Testsigma keeps running — this only stops Jenkins from waiting further (see the callout below).
Any other positive numberSame behavior, different ceiling. Raise this if you have a plan that legitimately runs longer than 4 hours.
0, unbounded, or noneNo ceiling at all. Polls purely on terminal status, indefinitely. The separate circuit breaker below still applies.
The failsafe does not stop Testsigma. If the 4-hour ceiling trips, poll-status.sh/.ps1 marks the Jenkins build FAILED and exits — it does not send a stop signal to Testsigma. Only clicking Abort in Jenkins does that (Part 9). This is intentional: a timeout might just mean your plan needed more time, not that it should be killed.
A second, unrelated safety mechanism: independent of the 4-hour failsafe, the script also fails the build if Testsigma's API itself doesn't respond successfully for 10 consecutive polls (~5 minutes at the default 30-second interval) — a dead-connection circuit breaker, not a TestPlan-duration check. A single successful poll resets that counter to zero, even if it just reports RUNNING.
A note on result values: Testsigma's polling endpoint reports a passing run as SUCCESS, not PASSED — the scripts in this guide already account for that, and separately check Testsigma's status field as a fail-safe: if a run is clearly finished but reports a result string that isn't recognized, the build fails with a clear error instead of waiting indefinitely. See the Reference section if you're curious how that's implemented.

Choosing TS_REPORT_LEVELS. This is a fixed value in the Jenkinsfile's environment {} block, marked 🔧 Customizable — not a per-build choice, since in practice almost everyone wants the same levels every run. The default in this guide is SUITE,CASE,STEP. Comma-separated, no spaces. PLAN is always fetched as a baseline regardless of what's listed here, so a report is never completely empty just because a deeper level's data wasn't available.

ValueWhat it fetchesIDs required

How SUITE and CASE find their own IDs, without you looking anything up: Testsigma's test_suite_results and test_case_results endpoints, called with no ID, return a paginated list of every suite/case result belonging to a run — the same shape the per-step results endpoint already used. fetch-report walks that list automatically once the run finishes and writes one file per result. This matters because a case or suite result ID doesn't exist until after the run has already started, so there was never a moment where you could type one into a form before triggering a fresh run — automatic discovery is the only way this could work cleanly.
Want to restrict a level to a specific subset instead of everything found for the run? That's still possible, just not through a build-time form field — add TS_SUITE_RESULT_IDS or TS_CASE_RESULT_IDS (comma-separated IDs) directly to the Jenkinsfile's environment {} block, right next to TS_REPORT_LEVELS. Most people never need this; it's there for the rare case where a run covers more ground than you want in every report.

Part 11 — Reading Your HTML Report

After fetch-report writes JSON (and .jsonl for STEP) into ts-reports/ in the job's workspace, generate-html-report converts all of it into one file: ts-report_<RunID>.html. The path is printed to the console as the last line of the build (see the [REPORT] line in Part 8's sample output) — open it directly from the agent's filesystem, or download it from the build's Artifacts list in Jenkins (it's archived there automatically). The report's masthead carries the Testsigma logo in the top-right corner — embedded as base64 so the report stays one self-contained file with no external image dependency — purely cosmetic, so the report reads as belonging to the product it's reporting on.

Every report opens with a collapsed Fetch Summary, above everything else. Click it open and it's a small table: one row per level, its status (OK, OK (baseline) for an implicitly-fetched PLAN, PARTIAL, or FAILED), and a plain-English reason — e.g. "3 case id(s) auto-discovered". It's collapsed by default since most runs fetch cleanly and this is a detail panel, not the headline — but its title line still says plainly if anything needs attention (e.g. "2 levels need attention"), so a real problem is never fully hidden behind a click.
The Test Plan Summary is a curated summary, not a raw dump. Testsigma's plan-level response has ~40 fields, most of which are internal execution config (step timeouts, hook IDs, internal run IDs) nobody needs at a glance. The section now leads with Result, Duration, Start/End Time, Triggered By, Build, and the pass/fail/stopped/not-executed counts — in plain English, with duration shown as e.g. 2m 36s instead of raw milliseconds. Everything else is still there, just collapsed into an "Additional execution details" section you click to expand (plain HTML <details>, no JavaScript) — nothing is deleted, it's just not the first thing you have to read past. The same plain-English headers and duration formatting apply to the Machine/Suite/Case/Step tables below it too.
v6: Test Step Results is now paginated per test case, not per row. Steps are grouped by testCaseId (the logical test case, not the per-execution caseResultId — so a retry that produces a second case result still lands in the same group) and each test case gets its own page: Prev/Next moves between whole test cases, one visible at a time, instead of stacking every test case's steps in one long scroll. Reference — Why This Works the Way It Does

Why not the official Testsigma Jenkins plugin? It has an unfixed security issue (CVE-2025-53661 — doesn't hide your API key properly), hasn't been updated since 2022, and can only trigger and wait — no abort, no result-level fetching, no report generation. This applies equally to Mac and Windows installs.

Why separate aborted/failure/success blocks instead of one shared step? Jenkins evaluates these in a fixed order (always first if present, then aborted/failure/success), and a single shared step running at the wrong point would either delete RUN_ID_FILE before the abort logic reads it, or generate the report before Testsigma's stop signal has actually gone out. Each outcome block now calls the shared fetchAndGenerateReport() function itself, at the point in its own logic where that's correct — after the stop signal for aborted, immediately for failure/success — then does its own cleanup.

Why was fetch/report moved out of post { always {...} }? An earlier revision put it there so a report would still be generated on FAILURE/STOPPED/timeout outcomes, not just success. That part was right — but Jenkins always runs the always block before aborted/failure/success, regardless of the order blocks are written in the file. That meant an aborted run's report was generated before the Testsigma stop signal went out, capturing a stale "RUNNING" snapshot instead of the run's true final "STOPPED" state. The fix: extract the fetch/generate/archive logic into a shared fetchAndGenerateReport() function (defined once, above the pipeline {} block) and call it explicitly from aborted/failure/success instead — for aborted specifically, after the stop signal. All three outcomes still get a report; the aborted one is just now correct.

Why does a fetch or report-generation failure not fail the build? The Jenkinsfile captures both scripts' exit codes via returnStatus: true and only logs a [WARN] line. A missing report is a reporting problem, not a testing problem — the build's pass/fail should keep reflecting what Testsigma actually did, not whether a downstream convenience script also succeeded.

Why does MAX_WAIT_MINUTES default to 240 instead of unbounded? A sensible failsafe beats an indefinitely hung Jenkins executor. 4 hours is a starting point, not a limit on what your TestPlan is allowed to need — raise it (or disable it) per build via the parameters form introduced in Part 7.

Why is the dead-connection circuit breaker a separate mechanism from MAX_WAIT_MINUTES? They detect different failures. MAX_WAIT_MINUTES asks "has the TestPlan been running too long?" — a question about Testsigma's execution. The circuit breaker asks "is Testsigma's API even answering?" — a question about connectivity, independent of how long the plan itself has been running. A single successful poll (any result, including RUNNING) resets the circuit breaker; it does not reset the MAX_WAIT_MINUTES clock.

Why does the script check both result and status to decide a run is finished, instead of just result? Testsigma's polling endpoint reports a passing run as SUCCESS, which is easy to miss if you only account for the more common PASSED convention used elsewhere in the same product. Rather than rely on a fixed list of known result strings being complete forever, the script treats status: STATUS_COMPLETED as the authoritative signal that a run has actually finished: if result is a recognized value, it reports pass/fail accordingly; if status says the run is done but result is something unrecognized, it fails the build loudly with a specific error rather than waiting indefinitely for a match that will never come. An unfamiliar API deserves a fail-fast default, not an infinite loop.

Why does fetch-report.sh avoid mapfile/readarray, and why is every array loop guarded with a length check? Both are compatibility choices for macOS specifically. mapfile is a bash 4+ builtin; macOS's system /bin/bash is permanently version 3.2 (Apple hasn't shipped a newer one since 2007, for licensing reasons) regardless of any newer bash a user may have installed separately, since the script's shebang resolves to the system binary. A plain while read loop achieves the same thing and has worked in every bash version since the 1990s. Separately, bash before 4.4 treats "${array[@]}" on a genuinely empty array as an unbound-variable error under set -u — a long-fixed quirk upstream that macOS's frozen 3.2 will never receive. Every loop over one of these arrays checks the length first (if [[ ${#array[@]} -gt 0 ]]) so the array is never expanded at all when empty, which avoids both the crash and a subtler issue: a bare fallback like "${array[@]:-}" stops the crash but iterates once with a blank value instead of zero times, which would trigger one malformed API call per run whenever a level genuinely finds nothing.

Why does the empty-array case matter in practice, not just in theory? A Test Plan with zero executed cases is a legitimate, if unusual, outcome — and the fetch logic needs to handle it the same way it handles the common case, without a crash or a spurious request standing in for "nothing to fetch."

Why can SUITE and CASE auto-discover their IDs, when a case/suite result ID doesn't exist until after a run has already started? Testsigma's documented API only shows single-ID lookups, but direct testing against a completed run showed that GET /test_runs/{run_id}/test_suite_results and GET /test_runs/{run_id}/test_case_results, called with no id, return a paginated list of every result for that run — undocumented, but confirmed working, in the same spirit as the undocumented-but-working stop endpoint below. fetch-report uses this by default and only falls back to requiring a manually-supplied ID if you've explicitly set one (as a subset filter) or if the call fails.

Why was TS_REPORT_LEVELS moved out of the build parameters entirely, when MAX_WAIT_MINUTES stayed? The two aren't the same kind of choice. MAX_WAIT_MINUTES is a per-run judgment call — how long is too long today might differ tomorrow. Which result levels to fetch is a one-time decision almost nobody revisits build to build: once you know you want Suite/Case/Step detail, you want it every time. Keeping it as a build parameter meant a field on every single "Build with Parameters" screen that virtually never changed and had nothing to fill in for the levels that auto-discover, which is exactly the confusion that motivated dropping it. It's still fully configurable — just as a one-line edit in the Jenkinsfile (Part 10) instead of a per-build form field.

Why was the MACHINE (per-device) level removed entirely instead of kept as an option? Testsigma's API doesn't confirm the same no-ID list support for machine results that SUITE and CASE have, so MACHINE never reached the same reliability — it needed a manual ID fallback that added a parameter for a level that, in practice, wasn't being used. Rather than keep a half-working, rarely-needed feature cluttering the setup, it was removed. The pattern that made SUITE/CASE reliable (list-without-id discovery) is exactly what a future MACHINE level would need to work the same way, if that's ever confirmed.

Why were TS_SUITE_RESULT_IDS and TS_CASE_RESULT_IDS removed from the build form instead of just made optional? Optional-but-present still shows up on every "Build with Parameters" screen as a field a person has to look at and decide to ignore. Since auto-discovery for these two is reliable, there's no realistic case where a first-time reader needs to fill them in — so they're gone from parameters {} entirely. The underlying capability wasn't removed from fetch-report itself; it just isn't wired to a build-time input anymore. If you personally need subset filtering, set the variable directly in the Jenkinsfile's environment {} block.

Why does the Test Plan Summary hide most of the raw response behind a collapsed section instead of showing everything flat? Testsigma's plan-level response has roughly 40 fields, and most of them are internal execution config (timeouts, hook IDs, internal run identifiers) that don't answer the question someone opening a report actually has. The fields shown up front — Result, Duration, Start/End Time, Triggered By, Build, and the pass/fail counts — are the ones that answer "did it pass and what happened," in the order a person would actually want them. Nothing is deleted — the rest is one click away in a native <details> element, which needs no JavaScript and works the same offline as everything else in this report.

Why one HTML file with JavaScript pagination instead of a page-per-level or page-per-20-rows split? A single path is what gets printed to the console and archived as a Jenkins artifact — there's exactly one thing to open, and the pagination still works fully offline since it's plain JS with no external dependency.

Why explicit size=100 and totalPages instead of trusting a "last page" flag? An earlier revision of this script relied on a boolean last field in the response to decide when to stop paging. Testing against real multi-hundred-step cases showed the more reliable signal is the page size you explicitly request plus the totalPages count the API returns for that request — so the script now asks for exactly 100 per page and loops until page reaches totalPages, with a defensive stop if a page ever comes back empty before that.

Why the real logo now, when an earlier revision deliberately used an original badge instead? The original badge avoided reproducing Testsigma's actual mark by default. Once you supplied Testsigma's logo directly and asked for it specifically, that concern doesn't apply the same way — this is your own internal tooling identifying the product it integrates with, which is a standard, low-risk use of a vendor's logo. It's embedded as base64 in both the guide and the generated report so nothing depends on an external file path that might not exist on the machine rendering it.

Why sh(script: 'cat ...') instead of reading the file directly in Groovy? Jenkins pipeline scripts run in a security sandbox that blocks raw Groovy/Java file reads unless an administrator approves each one. Routing through a shell command sidesteps this since shell commands are already allowed.

Why powershell(script: 'Get-Content ...') instead of reading the file directly in Groovy? Same sandbox restriction as on Mac — routed through PowerShell instead, which pipeline steps are already allowed to call.

Why pass the Run ID into the abort PowerShell block via withEnv() instead of Groovy string interpolation? The stop-call script is wrapped in Groovy triple-single-quotes so PowerShell's own $env: syntax isn't accidentally interpreted by Groovy first, so the Run ID is exported as an environment variable instead.
